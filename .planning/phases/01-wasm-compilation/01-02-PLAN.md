  <action>
  This task is an iterative compile-fix loop. The goal is `make configure && make build` completing with zero errors.

  **Strategy: Minimal module set first, then expand.**

  Step 1: Apply patches and run initial configure:
  ```bash
  cd /home/pooppoop/v9
  make configure 2>&1 | tail -50
  ```
  If configure fails, fix CMake issues in patches/edgejs-emscripten.patch or emscripten-toolchain.cmake.

  Step 2: Attempt first build:
  ```bash
  make build 2>&1 | tail -100
  ```

  Step 3: For each compilation error, determine the fix category:

  **Category A - Missing V8/Node type or function:**
  Add the missing declaration to the appropriate shim header based on layer:
  - Layer 0-2 (foundational): wasi-minimal-fixes.h, wasi-cppgc-stubs.h
  - Layer 3-5 (V8 internals): wasi-v8-internals.h, wasi-v8-essential-constants.h, wasi-namespace-fixes.h
  - Layer 6-8 (V8 API): wasi-v8-api-stubs.h, wasi-v8-isolate-extensions.h
  - Layer 9+ (complete types): wasi-v8-complete-types.h, wasi-v8-complete-missing.h, wasi-v8-node-missing.h
  - Catch-all: wasi-comprehensive-fixes.h, wasi-consolidated-fixes.h, wasi-additional-fixes.h

  **Category B - EdgeJS source needs modification:**
  Add a new hunk to patches/edgejs-emscripten.patch. Use `#ifdef EDGE_PLATFORM_EMSCRIPTEN` guards to conditionally compile Emscripten-specific code. Common patterns:
  - `#ifdef EDGE_PLATFORM_EMSCRIPTEN` to skip V8 JIT-related code
  - Stub out platform-specific syscalls (fork, execvp, raw sockets)
  - Replace native file I/O with Emscripten MEMFS calls where needed

  **Category C - Non-essential module has complex dependencies:**
  Stub the module. Create a minimal .cc file that registers the module name but has empty implementations. Non-essential modules for Phase 1 (stub these if they cause errors):
  - edge_tls_wrap.cc (TLS -- browser handles this)
  - edge_cares_wrap.cc (DNS c-ares -- browser handles this)
  - edge_tcp_wrap.cc (raw TCP -- not available in browser)
  - edge_udp_wrap.cc, edge_js_udp_wrap.cc (UDP -- not available in browser)
  - edge_pipe_wrap.cc (Unix pipes -- not in browser)
  - edge_tty_wrap.cc (TTY -- replaced by xterm.js in Phase 6)
  - edge_signal_wrap.cc (signals -- not in browser)
  - edge_spawn_sync.cc (sync spawn -- reimplemented in Phase 5)
  - edge_process_wrap.cc (process spawn -- reimplemented in Phase 5)
  - edge_worker_env.cc (worker threads -- not needed)
  - crypto/edge_crypto_bio.cc, crypto/edge_crypto_binding.cc, edge_crypto.cc (crypto -- replaced by Web Crypto in Phase 4)

  For each stubbed module, the approach is:
  - Do NOT remove the file from CMakeLists.txt (that requires more complex patching)
  - Instead, wrap the file's contents with `#ifndef EDGE_PLATFORM_EMSCRIPTEN` in the patch, and provide a minimal stub inside the `#else` block that satisfies the linker

  **Category D - Linker error (undefined symbol):**
  - If it's a V8 symbol: add stub to wasi-shims (likely wasi-v8-node-missing.h or wasi-v8-complete-missing.h)
  - If it's a libc/POSIX symbol: check if Emscripten provides it with different name/signature, or add to wasi-system-compat.h
  - If it's an N-API symbol: these are expected to be undefined (provided by JS imports at runtime). Verify `-sERROR_ON_UNDEFINED_SYMBOLS=0` and `--allow-undefined` are in link flags

  Step 4: After each fix, rebuild and check for new errors:
  ```bash
  cmake --build build -j$(nproc) 2>&1 | grep "error:" | head -20
  ```

  Step 5: Continue until `make build` exits with code 0 and produces dist/edgejs.wasm.

  Step 6: Run `make size` to report the binary size.

  **IMPORTANT constraints from user decisions:**
  - Per CONTEXT.md: "Consolidate before pushing forward" -- when adding new shim content, add to existing headers in the correct layer, do not create new fragmented headers
  - Per CONTEXT.md: "Patch files for EdgeJS modifications" -- all changes to edgejs-src/ go through patches/edgejs-emscripten.patch, never edit edgejs-src/ directly
  - Per CONTEXT.md: "Binary size: don't optimize yet" -- no size ceiling for Phase 1, get it compiling first
  - Per CONTEXT.md: "Stub non-essential modules" -- modules not in minimal set (fs, path, events, buffer, util, process) get stub implementations
  - Stay on Emscripten 3.1.64 unless it causes blocking issues that 4.0.23 would fix (per CONTEXT.md Claude's Discretion)

  <action>
  Create `tests/test-wasm-load.mjs` that verifies the .wasm binary loads and initializes correctly. Use Node.js with WebAssembly API to simulate browser loading (Node.js supports WebAssembly natively).

  The test must:

  1. **Load the .wasm binary directly via WebAssembly API:**
     ```javascript
     import { readFileSync } from 'node:fs';
     import { dirname, join } from 'node:path';
     import { fileURLToPath } from 'node:url';

     const __dirname = dirname(fileURLToPath(import.meta.url));
     const wasmPath = join(__dirname, '..', 'dist', 'edgejs.wasm');
     const wasmBuffer = readFileSync(wasmPath);
     ```

  2. **Verify the .wasm is valid WebAssembly:**
     ```javascript
     const valid = WebAssembly.validate(wasmBuffer);
     // assert valid === true
     ```

  3. **Compile the module (proves no compilation traps):**
     ```javascript
     const module = await WebAssembly.compile(wasmBuffer);
     ```

  4. **Check exported functions exist:**
     - Get module exports via `WebAssembly.Module.exports(module)`
     - Verify `_main` or `main` is in exports (or whatever the Emscripten entry point name is)
     - Verify `_malloc` and `_free` are in exports
     - Log all exports for debugging

  5. **Check module imports (what the N-API bridge must provide):**
     - Get module imports via `WebAssembly.Module.imports(module)`
     - Log import count and sample imports for debugging
     - Verify `env` namespace exists in imports (Emscripten convention)

  6. **Attempt instantiation with minimal stub imports:**
     - Create a Proxy-based import object that returns stub functions for any import
     - Try `WebAssembly.instantiate(module, stubbedImports)`
     - This may fail for complex modules -- that's OK for Phase 1. Log the error but don't fail the test on instantiation errors. The key assertion is that compile succeeds.
     - If instantiation succeeds, verify `instance.exports._malloc` is a function

  7. **Report binary size:**
     - Log raw size in MB
     - This is informational, not an assertion (per CONTEXT.md: no size ceiling for Phase 1)

  Use the same test harness pattern as test-basic.mjs (console.log PASS/FAIL, process.exit with count).

  Add a `test-wasm` target to the Makefile:
  ```makefile

  ```

  Update the `test` target to include test-wasm:

